# Git Commit Skill

Commit changes following conventional commit conventions with descriptive messages.

## Workflow

1. **Check repository state** (run in parallel):
   - `git status` to see modified/untracked files
   - `git diff` to see unstaged changes
   - `git diff --cached` to see staged changes
   - `git log --oneline -5` to understand commit message style

2. **Analyze changes**:
   - Review all modified files to understand what changed
   - Identify the type of change (feature, fix, refactor, chore, docs, test, style, perf, ci, build)
   - Determine the scope if applicable (e.g., component name, module)

3. **Stage files**:
   - Stage specific files by name rather than using `git add -A` or `git add .`
   - Never stage files that may contain secrets (.env, credentials, etc.)

4. **Create commit**:
   - Use conventional commit format for the title
   - **The title (subject line) must be completely in lowercase** — no uppercase letters; the body may use caps where needed for code identifiers (e.g. `GetPortfolio`)
   - Keep title under 72 characters
   - Add detailed body describing what and why (not how)
   - Use HEREDOC format for multi-line messages

## Commit Message Format

```
<type>[optional scope]: <short summary>

[optional body with more details]
```

### Types
- `feat`: New feature or functionality
- `fix`: Bug fix
- `refactor`: Code restructuring without changing behavior
- `chore`: Maintenance tasks, dependencies, tooling
- `docs`: Documentation only changes
- `test`: Adding or updating tests
- `style`: Formatting, whitespace (no code change)
- `perf`: Performance improvements
- `ci`: CI/CD configuration changes
- `build`: Build system or external dependencies

### Example commit command

```bash
CC_GIT_SKILL=1 git commit -m "$(cat <<'EOF'
refactor(auth): simplify token validation logic

- extract validatetoken() helper to reduce duplication
- remove unused tokencache class
- consolidate error handling in single try-catch block
EOF
)"
```

## Important Rules

1. **Never** add yourself as author or co-author
2. **Never** mention the commit was generated by Claude or AI
3. **Never** use `--no-verify` unless explicitly requested
4. **Never** commit files containing secrets or credentials
5. **Always** use specific file names when staging (not `git add .`)
6. **Always** verify the commit succeeded with `git status`
7. If pre-commit hooks fail, fix the issue and create a **new** commit (don't amend)
8. **Always** prefix the commit command with `CC_GIT_SKILL=1` — the global commit guard requires this marker to allow a skill commit. Without it the commit is blocked and redirected back to this skill.

## Node Version Issues

If the commit fails due to Node version mismatch (common in projects using nvm):
```bash
source ~/.nvm/nvm.sh && nvm use <required-version> && CC_GIT_SKILL=1 git commit ...
```

Check package.json or .nvmrc for the required Node version.
